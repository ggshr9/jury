Consensus code review

Multiple reviews.
One verdict.

Run your diff past a panel of LLM reviewers and see only the findings they agree on — ranked, clustered, and annotated with dissenting opinions. Built for teams who want AI review they can trust to gate a merge.

Try the CLI How it works

Open source. Apache 2.0. Bring your own API keys — nothing leaves your machine in the self-hosted setup.

The problem

Single-model review has a ceiling.

You've seen it. The first week the AI reviewer ships real bugs, the team celebrates. Week four, half the findings are bike-shedding variable names, and the one genuine SQL injection is buried at item 14. People hit --no-verify and the gate becomes decorative.

This isn't a prompt-engineering failure. It's a structural one. A single model has one opinion about what matters in a diff, with one set of training blind spots. You can't tune your way out of it — tighter threshold loses real bugs, looser threshold drowns you in noise.

“The underlying model is fine. The filter is the product.”

How it works

A panel, not a pundit.

1 · Fan out

Your diff goes to 3–5 role-specialized reviewers in parallel. Each uses a different model, a different prompt, different training data.

→
2 · Cluster

Findings that point at the same underlying issue get merged across reviewers, even when worded differently. File + line + category exact match, then embedding similarity for semantic dupes.

→
3 · Quorum

Consensus rules decide what makes the verdict. Unanimous for critical, majority for important, any for minor. Every default is tunable.

→
4 · Verdict

Short, ranked, trustable. Informed dissent surfaces as a separate section — minority opinions worth reviewing, not noise to ignore.

Sample verdict

VERDICT · 4 findings · 2 critical · 1 important · 1 dissent worth review

CRITICAL  4/4 agreed    auth.ts:42
  SQL injection: userId reaches db.query() without sanitization
  ├─ bug-hunter      (claude-sonnet-4.6): flagged direct interpolation
  ├─ security        (gpt-5.4): recommends parameterized query via prepare()
  ├─ concurrency     (deepseek-r1): flagged as both SQL injection and race
  └─ maintainability (qwen-coder):  same injection pattern

CRITICAL  4/4 agreed    worker.ts:78
  Race condition: shared sessions map mutated without lock across goroutines

IMPORTANT 3/4 agreed    auth.ts:120
  Missing null check on decoded.email before .toLowerCase()

DISSENT   1/4, elevated utils.ts:15
  concurrency: memoize cache is not thread-safe; reviewed evidence shows
  two concurrent call paths. Others didn't flag — surfaced because category
  = race-condition AND evidence present.

Configuration

A single file.

The jury.yaml at your repo root is the source of truth. Switch reviewers, adjust quorum rules, short-circuit tiny PRs, apply per-path overrides. The CLI, the GitHub Action, and the pre-commit hook all read from the same file. Everyone on your team sees the same verdict because everyone's running the same config. 

version: 1

reviewers:
  - name: bug-hunter
    model: claude-sonnet-4.6
  - name: security
    model: gpt-5.4
    role: security
  - name: concurrency
    model: deepseek-r1
    role: concurrency

quorum:
  critical: unanimous
  important: majority

dissent:
  elevate_categories: [race-condition, auth, injection]
  weak_categories:    [naming, style]

routing:
  - paths: ["auth/**", "crypto/**"]
    override_quorum:
      important: unanimous

Full reference, including reviewer roles, model catalog, and per-path routing rules, in docs/dsl.md.

Quick start

Three commands.

  1. 1

    Install the CLI

    $ curl -sSL jury.dev/install | sh
  2. 2

    Configure your panel

    $ jury init  # interactive; writes jury.yaml
  3. 3

    Review your branch

    $ jury review main..HEAD

Or drop a GitHub Action into .github/workflows/ and let it post inline PR comments.

Is it for you?

We'll tell you when it isn't.

Worth the cost if

  • You gate merges on AI review and false positives are killing trust
  • Your code is security-critical — auth, crypto, data handling
  • Your service is concurrency-heavy and single-model review misses races
  • You're an OSS maintainer drowning in contributor PRs
  • You've tried CodeRabbit / Qodo / Greptile and turned them off

Stick with one model if

  • Your PRs are mostly doc / dependency bumps
  • You have strong static tooling (Rust + clippy, type-strict TS) already
  • 4× API cost isn't justifiable for marginal improvement
  • You don't run AI review as a merge gate, only as advisory
  • You're doing a throwaway prototype

Read the full breakdown in docs/use-cases.md.

Status

Being built in the open.

Jury is pre-v0.1. The architecture described here is implemented incrementally, with an honest validation gate before we ship.

  • Experiment 0 — validation on 30 historical bug-introducing commits. If ensemble gains don't beat single-model by 15+ pp Top-3 precision, we walk away from the idea.
  • CLI scaffold — fixed 3-model panel, no DSL yet
  • Consensus engine — clustering + quorum + dissent
  • jury.yaml v1 parser + validator
  • GitHub Action
  • Reviewer adapter plugins (Codex CLI, Claude, OpenRouter, Ollama)
  • Pre-commit hook
  • IDE plugins

No vapor ETAs. Milestones post when they ship.
